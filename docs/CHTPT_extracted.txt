
Thiết kế hệ thống soạn thảo văn bản cộng tác
Hướng thiết kế dùng React + TipTap + Yjs + Socket.IO + Node.js + Express + MySQL, trong đó nội dung rich text được lưu bằng JSON snapshot.

3. Cấu trúc dữ liệu MySQL theo hướng lưu JSON
Do hệ thống hỗ trợ văn bản định dạng giàu, tài liệu không chỉ lưu plain text. Mỗi document nên lưu nhiều biểu diễn khác nhau để phục vụ các mục đích riêng.
Trường
Kiểu dữ liệu
Mục đích
content_text
LONGTEXT
Dùng cho preview, tìm kiếm, hiển thị danh sách tài liệu.
content_json
JSON
Lưu cấu trúc rich text của TipTap, giữ định dạng như heading, bold, align, list.
content_html
LONGTEXT
Dùng cho hiển thị nhanh, export hoặc render phía server nếu cần.
ydoc_state
LONGBLOB
Lưu trạng thái Yjs để phục hồi realtime collaboration chính xác.

3.1. Bảng users
CREATE TABLE users (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE,
  password_hash VARCHAR(255),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
3.2. Bảng documents
CREATE TABLE documents (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  title VARCHAR(255) NOT NULL,
  content_text LONGTEXT,
  content_json JSON,
  content_html LONGTEXT,
  ydoc_state LONGBLOB,
  owner_id BIGINT NOT NULL,
  current_version INT DEFAULT 1,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (owner_id) REFERENCES users(id)
);
3.3. Bảng document_collaborators
CREATE TABLE document_collaborators (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  role ENUM('owner', 'editor', 'viewer') NOT NULL DEFAULT 'editor',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY unique_document_user (document_id, user_id),
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
3.4. Bảng document_versions
CREATE TABLE document_versions (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  version_number INT NOT NULL,
  content_text LONGTEXT,
  content_json JSON,
  content_html LONGTEXT,
  ydoc_snapshot LONGBLOB,
  created_by BIGINT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
  FOREIGN KEY (created_by) REFERENCES users(id)
);
3.5. Bảng document_updates
CREATE TABLE document_updates (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  document_id BIGINT NOT NULL,
  user_id BIGINT,
  update_data LONGBLOB NOT NULL,
  client_id VARCHAR(100),
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (document_id) REFERENCES documents(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id)
);
Bảng document_updates dùng để lưu các Yjs update nhỏ. Bảng documents và document_versions lưu snapshot đầy đủ để load nhanh, restore version và giảm chi phí replay quá nhiều update.
4. API HTTP cần chỉnh khi lưu JSON
Khi dùng TipTap, API không nên chỉ nhận trường content dạng chuỗi. API nên nhận và trả về contentText, contentJson, contentHtml và ydocState.
4.1. Auth/User
POST /api/auth/register
POST /api/auth/login
GET  /api/users/me
4.2. Document
GET    /api/documents
POST   /api/documents
GET    /api/documents/:id
PUT    /api/documents/:id
DELETE /api/documents/:id
Ví dụ response khi mở tài liệu:
{
  "id": 1,
  "title": "Báo cáo hệ thống phân tán",
  "contentText": "Hello world",
  "contentJson": {
    "type": "doc",
    "content": [
      {
        "type": "paragraph",
        "content": [
          { "type": "text", "text": "Hello world" }
        ]
      }
    ]
  },
  "contentHtml": "<p>Hello world</p>",
  "version": 3,
  "role": "editor"
}
Ví dụ request khi lưu tài liệu:
{
  "title": "Báo cáo hệ thống phân tán",
  "contentText": "Hello world edited",
  "contentJson": { "type": "doc", "content": [] },
  "contentHtml": "<p>Hello world edited</p>",
  "ydocState": "base64-encoded-yjs-state"
}

WebSocket không gửi toàn bộ JSON sau mỗi lần gõ. Realtime chỉ nên gửi Yjs update nhỏ. JSON/HTML/Text snapshot dùng cho load lần đầu, autosave, version và restore.

5.1. Ví dụ payload
Client join document:
socket.emit("join-document", {
  documentId: 1,
  user: { id: 2, name: "Binh", color: "#2f80ed" }
});
Client gửi Yjs update:
socket.emit("yjs-update", {
  documentId: 1,
  userId: 2,
  update: Array.from(update),
  clientId: "client_abc_123"
});
Server broadcast update cho client khác trong cùng room:
socket.to(`document:${documentId}`).emit("yjs-update", {
  documentId,
  userId,
  update,
  clientId
});

Các lệnh lưu snapshot ở frontend:
const payload = {
  title,
  contentText: editor.getText(),
  contentJson: editor.getJSON(),
  contentHtml: editor.getHTML(),

Khi mở tài liệu từ API, frontend có thể phục hồi nội dung bằng contentJson nếu chưa dùng ydoc_state:
editor.commands.setContent(document.contentJson);
7. Quy tắc lưu dữ liệu
Khi người dùng gõ, client không gửi toàn bộ JSON mà gửi Yjs update qua Socket.IO.
Server kiểm tra quyền editor/owner trước khi chấp nhận update.
Server broadcast Yjs update cho các client khác trong cùng document room.
Autosave định kỳ 5-10 giây lưu content_text, content_json, content_html và ydoc_state vào MySQL.
Khi tạo version, server lưu snapshot vào bảng document_versions.
Khi restore version, server cập nhật lại documents và broadcast sync-state-response cho các client đang mở.

9. Offline editing
Khi mất mạng, client hiển thị trạng thái Offline nhưng vẫn cho editor tiếp tục chỉnh sửa local.
Yjs giữ thay đổi local; có thể kết hợp IndexedDB để không mất dữ liệu khi reload.
Khi reconnect, client join lại document room và gửi các pending updates.
Server merge update bằng Yjs rồi gửi sync-state-response để các client đạt cùng trạng thái.
10. Kịch bản demo
User A đăng nhập và tạo tài liệu mới.
A mời User B với quyền editor và User C với quyền viewer.
A và B cùng chỉnh sửa văn bản rich text: heading, bold, căn giữa, bullet list.
C mở tài liệu, xem được nội dung nhưng không chỉnh sửa được.
A và B nhìn thấy cursor/vùng chọn của nhau.
B mất mạng, tiếp tục sửa offline, sau đó reconnect và đồng bộ lại.
A tạo version, xem lịch sử và restore một version cũ.
Reload trang, nội dung và định dạng vẫn được lưu từ content_json/ydoc_state.
11. Kết luận thiết kế
Sau khi chuyển sang hướng lưu JSON, hệ thống cần xem content_json là snapshot chính cho rich text, content_text là bản text phục vụ preview/tìm kiếm, content_html là bản render nhanh, còn ydoc_state là trạng thái cộng tác realtime. Cách tách này giúp hệ thống vừa hỗ trợ định dạng văn bản đầy đủ, vừa đảm bảo đồng bộ phân tán bằng Yjs.
